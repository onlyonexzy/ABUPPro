# GDTTools 原型项目说明

## 项目定位

这是一个用于原型展示、需求评审和流程走查的前端静态项目。
当前版本重点用于展示登录、工具工作台、GBF 转化以及新的刷写模块入口。

注意：

- 这不是正式交付版本
- 没有真实后端接口
- 没有真实 ECU 通信或刷写能力
- 主要用于界面评审和流程占位

## 运行方式

项目无需安装依赖，也不需要启动本地服务。

直接使用浏览器打开根目录下的 `index.html` 即可预览。

页面依赖外部 CDN 加载 `Font Awesome` 和 `Google Fonts`，如果网络受限，图标或字体可能加载不完整，但不影响主要原型浏览。

## 当前模块

### 1. 登录与状态模拟

包含在线登录、域账号登录、离线登录、失败锁定、网络切换、数据版本状态和超时退出等原型能力。

### 2. 工具工作台

登录后进入桌面式工作台，支持多窗口切换和基础工具菜单展示。

### 2.1 控制台（已重构）

控制台页面重新设计为汽车诊断工具风格的总线拓扑管理界面：

- **左侧面板**：总线/ECU 树形视图，支持展开/折叠、勾选/取消、节点选中
  - 顶部工具栏：添加总线、添加节点、删除、编辑、DBC/LDF 导入
  - 树形层级：总线（CAN/Ethernet/LIN）→ 节点（ECU/Slave/Master）
  - 不同总线类型使用不同颜色图标区分（CAN 红色、Ethernet 绿色、LIN 紫色）
  - 点击总线或子节点会联动切换右侧拓扑图和底部 Tab
- **可拖拽分割条**：左右面板宽度可调，支持折叠/展开左侧面板
- **右侧面板**：网络拓扑图区域
  - 顶部信息栏：显示当前总线类型、波特率、IP 等参数
  - 中部画布：ECU 节点与总线接口的可视化连接拓扑
    - 单 ECU：简单左右连线
    - 多 ECU：2 列网格布局，共享水平干线连接到总线接口
    - 画布可滚动，支持大量节点展示
  - 底部 Tab 栏：CAN1 / Ethernet1 / LIN1 总线切换
- Mock 数据：CAN1（ECM/TCU/ABS/BCM/SRS）、Ethernet1（GW/IVI/TBOX/ADAS）、LIN1（Master/Slave 1-3）
- 样式独立于 `assets/css/console.css`，交互逻辑在 `assets/js/console-diagram.js`

### 3. GBF 转化

保留原有的 GBF 转化原型，用于展示文件列表、Header 信息、结果区和整车打包等交互。

文件列表左侧按“原始文件”和“GBF 文件”分组展示，两个分组标题均支持点击收起和展开，便于在文件较多时聚焦查看其中一类文件。

### 4. 单件刷写

旧刷写逻辑已移除，当前重新创建为占位页面，并由 `flash-config.js` 与 `flash-config.css` 单独负责。
包导入、配置、执行、日志和结果等能力均为“暂定”。

刷写配置初始数据中，`CAN1` 与 `CANFD1` 的 ECU 会同步在 `ETH1` 下生成一份镜像数据，镜像 ECU 使用 `ETH1` 的以太网参数模板（逻辑地址、功能地址、IP、端口等）展示，便于对比不同总线下的配置效果。
对于这些镜像 ECU，列表中会在 `ECU（供应商）` 右侧显示来源小标签（`CAN` 或 `CANFD`），用于区分复制来源。
镜像 ECU 的 `IP地址` 固定不展示具体值（显示为 `--`），`刷写类型` 保持与复制前源 ECU 一致。
`ECU参数` 中原 `XOTA逻辑地址` 文案调整为 `逻辑地址`，并将原 `OBD连接刷写方式` 调整为 `打开网关直连通道` 勾选项。
`打开网关直连通道` 仅在 `ETH` 总线且 ECU 为非 `CAN/CANFD` 镜像复制数据时显示。
#### 单件刷写窗口（新增）

已新增独立 `single-flash.js` 窗口，入口位于左侧导航“统一刷写 > 单件刷写”，并放置在“刷写配置”下方：

- 左侧显示 ECU 列表，直接复用刷写配置中的 ECU 数据源（移除了左上角操作按钮）
- 右侧顶部提供按钮：获取ECU信息、刷写、上传LOG
- 右侧 ECU 标题下新增“ECU通讯参数”条：`ETH` 显示`逻辑地址/功能地址/IP地址`，`CAN/CANFD` 显示`请求地址/响应地址/功能地址`
- 通讯参数条右侧新增“设置”按钮，点击后会打开并置顶“刷写配置”窗口，并自动定位到当前选中的 ECU
- 中部展示刷写信息表（包名、类型、大小、版本、序列号、次数、间隔）
- 底部展示 LOG 数据滚动区

#### 刷写策略界面（已重构）

刷写策略（`flash-config-ecu-strategy.js`）采用垂直卡片队列布局，替代原横向链路流程图：

- **A 区（诊断流程）**：紧凑配置区，流程模式 radio 与对应控件在同一行
- **B 区（刷写包队列）**：垂直卡片列表，每张卡片内联 GBF 信息和策略字段（刷写次数、间隔），卡片间有向下箭头连接，支持上移/下移/移除；队列可重复引用同一主包或已导入包，重复版本导入时不新建模板而是直接追加队列项
- **C 区（循环策略）**：整体循环次数 + 执行预览文本 + 确认按钮

### 7. 系统设置 — 统一刷写设置（已完成）

入口位于左侧导航"系统设置 > 统一刷写设置"，采用上方 Tab + 下方内容布局：

- **顶部导航**：5 个功能 Tab 页签，点击切换下方内容
- **右侧内容**：每个 Tab 对应独立功能卡片

#### Tab 1：地址设置
- 可编辑 Tester 地址（仅 IP 地址）
- 可编辑网关地址（逻辑地址 + IP 地址）
- 提供保存按钮并进行基础格式校验

#### Tab 2：日志上传
- 分"单件刷写"和"整车刷写"两组独立配置
- 每组仅保留：日志上传方式（手动/自动）
- 默认值：手动上传日志

#### Tab 3：DID 配置
同一页面内分两个子区块：
- **单件刷写 DID**：表格管理，默认内置 4 条固定解析 DID，支持添加（4 位 HEX 校验）和删除（二次确认）
- **整车刷写 DID**：表格管理逻辑地址 + DID 组合，支持添加和删除，默认预置网关 VIN 读取记录

#### Tab 4：数据版本
- 展示当前数据版本和云端最新版本
- "检查数据"按钮模拟异步获取云端版本
- 版本一致时"立即更新"禁用，版本不同时允许一键更新

#### Tab 5：文件存储
- 配置下载文件路径和日志文件路径
- 路径修改立即生效，影响 GBF 转化下载和刷写日志存储

样式文件：`assets/css/settings.css`，交互逻辑：`assets/js/settings.js`

### 5. 工程诊断

工程诊断模块包含四个子功能：基础诊断、快捷诊断、OBD诊断、PDX校验。

#### 5.1 基础诊断（已完成）

入口位于左侧导航"工程诊断 > 基础诊断"，采用经典 DVR 工具风格布局：

- **左侧面板**：总线/ECU 树形视图，复用控制台 `busConfig` 数据
  - CAN1（ECM/TCU/ABS/BCM/SRS）、Ethernet1（GW/IVI/TBOX/ADAS）、LIN1（Master/Slave 1-3）
  - 点击 ECU 选中后，右侧加载该 ECU 的诊断环境
  - 面板可折叠/展开

- **右侧主区域**：顶部诊断地址栏 + "服务"/"PDX校验"两个功能页签
  - 地址栏：CAN 显示请求/响应地址，ETH 显示逻辑地址/IP，TP 模式可选
  - **服务页签**（DVR 风格左右分栏）：
    - 左侧：自定义发送入口 + UDS 服务列表，10 个基础服务（10/11/14/19/22/27/28/2E/31/85）
    - 中间：参数配置面板（子功能 Radio、DID 多选、DTC 分组等，根据服务动态渲染）+ HEX 请求输入 + 响应解析树
    - 自定义入口：不显示具体 UDS 服务参数面板，仅由用户自行填写 HEX 请求数据，保留功能寻址勾选、响应区和右侧收发日志布局
    - 右侧：收发日志表（序号/时间/方向/数据）
  - **PDX校验页签**：DTC 故障码表 + DID 列表表 + 一键读取

样式文件：`assets/css/basic-diag.css`，交互逻辑：`assets/js/basic-diag.js`

#### 5.2 快捷诊断（占位）

入口位于左侧导航"工程诊断 > 快捷诊断"，当前为占位页面，后续设计。

#### 5.3 OBD诊断（占位）

入口位于左侧导航"工程诊断 > OBD诊断"，当前为占位页面，后续设计。

#### 5.4 PDX校验

### 6. 整车刷写

整车刷写模块由 `vehicle-flash.js` 与 `vehicle-flash.css` 负责，支持串行/并行两种刷写模式。

整车刷写左侧面板仅保留"串行/并行"模式切换，并行配置已移至**刷写配置**模块。

#### ECU 列表数据源

整车刷写的 ECU 列表和执行顺序统一由**整车刷写顺序配置**弹框（`flash-config-sequence.js` / `FlashConfigSequenceModule`）驱动：

- **串行模式**：调用 `FlashConfigSequenceModule.getSerialExecList()` 获取用户在"串行顺序"Tab 中配置好的有序 ECU ID 列表，按此顺序渲染 ECU 表格。若用户未配置（返回空列表），则降级为 snapshot 原始遍历顺序。
- **并行模式**：调用 `FlashConfigSequenceModule.getParallelSteps()` 获取"并行配置"Tab 中的步骤→网段→ECU 结构，执行顺序区域展示网段卡片，ECU 列表按步骤/网段分组。若用户未配置（返回空列表），则降级为 `FlashConfigParallelModule` 的总线分组逻辑。
- ECU 的详细数据（版本号、供应商、刷写包等）始终从 `FlashConfigShared.getSnapshot()` 中获取。

#### 并行配置（弹窗，位于刷写配置）

并行配置入口位于**刷写配置**页面左侧工具栏（`<i class="fa-solid fa-shuffle"></i>` 图标按钮）。点击后弹出居中弹窗（`flash-config-parallel.js`），独立管理并行编排状态：

- **Pre / Post 脚本配置**：支持"默认脚本"下拉和"自定义脚本"本地文件两种模式
- **执行步骤编排**：
  - 总线按步骤列横向展示，同一步骤内**并行**，不同步骤**串行**
  - 每个总线卡片内 ECU 列表为串行顺序，支持拖拽调整
  - 每个 ECU 行包含"终止FA"复选框；弹窗顶部提供"一键终止所有FA"全局开关
  - 修改步骤编号可动态调整总线分组
- 保存后配置通过 `FlashConfigParallelModule.getSavedConfig()` 对外暴露，作为未配置整车刷写顺序时的降级数据源

## 打包为单个 HTML 文件

项目支持通过 `build.js` 脚本将所有本地资源（CSS、JS、图片）内联到一个 HTML 文件中，方便直接发送给他人查看。

### 前置条件

已安装 Node.js（任意版本均可，脚本使用纯内置模块，无需 npm install）。

### 基本用法

```bash
# 默认打包 — CDN 图标/字体保留，查看时需联网
node build.js

# 全离线打包 — CDN 资源也下载内联，无需联网即可查看完整效果
node build.js --offline
```

### 可选参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--input` | `index.html` | 指定入口 HTML 文件 |
| `--output` | `GDTTools-standalone.html` | 指定输出文件名 |
| `--offline` | 无 | 启用后 CDN 资源也会下载并内联 |

### 工作原理

脚本自动扫描 HTML 中的所有资源引用（不硬编码文件路径），未来新增 CSS/JS/图片只要在 HTML 中正常写了引用标签，打包时会自动识别并内联：

- `<link rel="stylesheet" href="...">` → 内联为 `<style>`
- `<script src="...">` → 内联为 `<script>`
- `<img src="...">` → 转为 base64 data URI
- CSS 中的 `url(...)` → 转为 base64 data URI

### 输出文件

打包后生成的 HTML 文件可直接双击打开，也可通过邮件、微信、钉钉等发送给他人。
